1.2 How do we visualize spatial data?

A map is a special type of data visualization for spatial data but it isn‚Äôt the only one.

Plots and tables are other common ways to visualize spatial data.

Visuals can be further transformed using animation and interactivity.

Maps, graphics, and tables also vary in format: print, web, mobile, etc.

2 What is the ‚Äúgrammar of graphics‚Äù?

Grammar of graphics

From The Grammar of Graphics for Introduction to data visualisation with ggplot2 QCBS R Workshop Series.

Concept from Grammar of Graphics book (1999), from Leland Wilkinson.

Grammar of graphics

  • Data
  • Mapping
  • Statistics
  • Scales
  • Geometries
  • Facets
  • Coordinates
  • Theme

Introduction to the ‚Äúgrammar of graphics‚Äù adapted from a 2020 ggplot2 workshop (part 1) by Thomas Lin Pedersen.

2.1 Data

  • Structure and representation of data determines what you can and can‚Äôt do with it
  • Data is expected to be in a ‚Äútidy‚Äù format (a.k.a. long format data)

A few definitions:

  • variable: a quantity, quality, or property that you can measure.
  • value: the state of a variable when you measure it. The value of a variable may change from measurement to measurement.
  • observation: a set of measurements made under similar conditions
  • tabular data: a set of values, each associated with a variable and an observation.

Excerpted from Ch. 2 of R for Data Science.

What works best with the grammar of graphics is tidy data:

  • each value is placed in its own ‚Äúcell‚Äù,
  • each variable in its own column, and
  • each observation in its own row.

2.2 Mapping

  • Mapping allows data to be understood by the graphics system

    • Aesthetic mapping: variables -> graphical properties in the geometry
    • Facet mapping: variables -> panels in layout

2.3 Statistics

  • Data may not represent the displayed values

  • Statistics transform input variables into displayed values, e.g.

    • Counting the number of observations by category
    • Calculating summary statistics for a boxplot

Statistics can be used prior to plotting data or used by a plotting function directly.

2.4 Scales

  • Scales translate between value ranges and graphical properties, e.g.

    • Categories -> Colors
    • Numbers -> Position

  • Scales use a specific type of interpolation, e.g. discrete, continuous, etc., so not all scales work with all variables.

In the not too common case where data directly represents a graphical property, e.g. a ‚Äúcolor‚Äù column, you can use a special type of scale called an ‚Äúidentity‚Äù scale.

2.5 Geometries

  • How translate aesthetics into graphical representations

  • Basic geometries (e.g. points, lines, polygons) can be combined into more complex geometries (e.g. box plot, map)

  • Typically, the geometries are the same as the ‚Äútype‚Äù of plot

2.6 Facets

  • Defines how data is split across multiple panels

2.7 Coordinates

  • Positional aesthetics must be interpreted by a coordinate system

  • Defines the physical mapping of aesthetics to the paper

We usually think about the cartesian coordinate system‚Äîbut there are a lot of different kinds of coordinate systems.

2.8 Theme

  • Every other part of the graphic that isn‚Äôt linked to the data:

    • Fonts
    • Spacing
    • Colors
    • Weights
    • etc.

3.6 Add aesthetics and layers

Set mapping with aes()

Next, we can set the mapping using the aes() function. We can start by mapping long (longitude) to x and lat (latitude) to y:

ggplot(
  data = storms,
  mapping = aes(
    x = long,
    y = lat
  )
)

Before we can see any data, we need to tell ggplot() how we want to represent the data using layers.

The mapping argument of the ggplot() function defines how variables in your dataset are mapped to visual properties (aesthetics) of your plot.

Note that the storms data is not spatial at this point.

Layers in ggplot2 are most often added using ‚Äúgeoms‚Äù (short for geometry) functions like geom_point(), geom_col(), or geom_histogram().

Let‚Äôs give it a try using geom_point():

ggplot(
  data = storms,
  mapping = aes(
    x = long,
    y = lat
  )
) +
  geom_point()

You can define the data and mapping parameters globally using ggplot() or locally for a single geom function. This code does the exact same thing as the prior block:

ggplot() +
  geom_point(
    mapping = aes(
      x = long,
      y = lat
    ),
    data = storms
  )

What is this + operator?

Notice that we are combining the ggplot() and geom_point() functions using the + operator.

This is a style of function or operator known as an infix operator.

This operator works is similar to the pipe operator, %>% or |> that passes the output from one function (or line of code) to the next line as the first parameter in a following function.

For example, see how we can use a series of pipes to operate on the storms data:

storms |>
  distinct(name)
#> # A tibble: 268 √ó 1
#>   name    
#>   <chr>   
#> 1 Amy     
#> 2 Blanche 
#> 3 Caroline
#> 4 Doris   
#> 5 Eloise  
#> 6 Faye    
#> # ‚Ñπ 262 more rows
storms |>
  distinct(name) |>
  nrow()
#> [1] 268

The prior block is the same as the following block:

storm_names <- distinct(storms, name)
nrow(storm_names)
#> [1] 268

Using pipes can help avoid the need for intermediate objects like storm_names that might only exist to move the output of one function to the input for another function.

The first parameter for ggplot() is data and the first parameter for all geom functions is mapping so you often see these passed as unnnamed parameters:

ggplot(storms) +
  geom_point(
    aes(
      x = long,
      y = lat
    )
  )

See how the plot is now showing axis labels.

You can even leave off the x and y parameter names:

ggplot(storms) +
  geom_point(aes(long, lat))

Function parameters can be specified by position as well as name.

Mapping variables are not strings!

The variables passed to aes() don‚Äôt have quotes around them! This is because aes() is a data-masking (also known as a quoting function) where the inputs are evaluated in the context of the data.

TLDR: only put quotes around your aesthetic mappings when you want to pass a literal string.

Next we can map one of the variables (status or storm classification) to an aesthetic (color):

ggplot(storms) +
  geom_point(aes(long, lat, color = status))

Note that we now have a legend, added automatically when we defined a mapping to color. Since color is not a ‚Äúpositional aesthetic‚Äù we can‚Äôt easily interpret the meaning of the graphic without a legend, labels, or annotations.

In addition mapped aesthetics, we can set ‚Äúfixed‚Äù aesthetics (called fixed because the same value is used for every observation in the plot).

When you have a lot of overlapping features (known as overplotting), you may want to reduce the alpha (or transparency) of the features:

ggplot(storms) +
  geom_point(
    aes(
      x = long,
      y = lat,
      color = status
    ),
    alpha = 0.4
  )

Fixed aesthetics should be defined for each geom‚Äîthey are ignored if you pass them to ggplot().

Aesthetics can also be defined using a function. In this example, we are using the boolean operator == to compare the values from ‚Äústatus‚Äù to the string ‚Äúhurricane‚Äù:

ggplot(
  data = storms,
  mapping = aes(
    x = long,
    y = lat,
    color = status == "hurricane"
  )
) +
  geom_point()

4 Visualize spatial data

To start, let‚Äôs convert our storms data set from a data frame into a sf object using sf::st_as_sf():

storms_sf <- sf::st_as_sf(
  storms,
  coords = c("long", "lat"),
  crs = 4326
)

Always supply coordinates with longitude before latitude!

sf::st_as_sf() requires that you supply coordinates in lon lat order. This is also the standard order for GeoJSON files, Shapefiles, and KML files. Check out lon lat lat lon by Tom MacWright for more information.

Finally, we can look at these observations on a map:

ggplot(data = storms_sf) +
  geom_sf(aes(color = status))

That first map may not look much different.

But converting the data to a sf object allows us to use coord_sf() to transform the coorindate reference system on the fly:

ggplot(data = storms_sf) +
  geom_sf(aes(color = status)) +
  coord_sf(crs = "EPSG:3035")

This is equivalent to converting the coordinate reference system in advance using st_transform():

storms_sf |>
  st_transform(2804) |>
  ggplot() +
  geom_sf(aes(color = status))

Note, that this is another example of how we can use the pipe (|>) to move data between functions in R.

However, to really show the potential, we need some more data. Install and load the rnaturalearth package and then load data for the coastline and countries:

# pak::pkg_install(c("rnaturalearth", "rnaturalearthdata"))
library(rnaturalearth)

coastline <- ne_coastline()

countries <- ne_countries()

Put them together and we are starting to get somehwere:

ggplot() +
  geom_sf(data = storms_sf, aes(color = status)) +
  geom_sf(data = coastline, color = "white")

But the map is zoomed out to show the whole world‚Äînot just the north Atlantic storm observations. We can use the xlim and ylim parameters of coord_sf() to ‚Äúzoom‚Äù in on a smaller area:

storms_bbox <- storms_sf |>
  st_transform("EPSG:3035") |>
  st_bbox()

storms_map <- ggplot() +
  geom_sf(data = countries, fill = "white") +
  geom_sf(data = storms_sf, aes(color = category), alpha = 0.3) +
  geom_sf(data = coastline, color = "black") +
  coord_sf(
    xlim = c(storms_bbox$xmin, storms_bbox$xmax),
    ylim = c(storms_bbox$ymin, storms_bbox$ymax),
    crs = "EPSG:3035"
  )

storms_map

What makes a bad map bad?

For new map makers who might not know any better, it is often:

  • Confusing or distracting layout
  • Not enough color contrast for features
  • Cluttered labels, legends, or features

What makes a fine map just fine?

For more experienced map makers who might be in a hurry: - Not enough thought on font sizes and styling - No feature generalization (simplifying geometry where appropriate for the map scale or subject) - Inappropriate color scales

Other related considerations

There are many cartographic considerations that also apply to other all types of data visualization:
